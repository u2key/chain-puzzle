# ソフトウェア要求仕様書 (Software Requirements Specification)
## 物理演算ベース・なぞり消しパズルゲームシステム

## 1. システム概要
本システムは、スマートフォンの縦画面サイズを基準とし、PCやタブレットなどのマルチデバイスでもブラウザ上でプレイ可能な、物理演算を用いた落ち物パズルゲームである。
画面上部から降り注ぐ円形のオブジェクトを、プレイヤーが一筆書きのようになぞって3つ以上繋げて消去し、制限時間60秒以内での高得点を目指す。ゲーム終了後は、獲得スコアとユーザー識別子を非同期通信でサーバーに送信・保存し、最新のトップ10ランキングを表示する競争型のWebアプリケーションである。

---

## 2. システム構成・技術スタック
開発効率およびクロスプラットフォーム対応の最大化を目的とし、以下のWeb標準技術およびオープンソースライブラリを採用する。

* クライアントサイド
  * 言語: HTML5, CSS3, JavaScript (ES6以降)
  * ゲーム/物理エンジン: Matter.js（2D物理演算）および Phaser 3（またはCanvas APIを用いたカスタム描画フレームワーク）
  * 通信方式: Fetch API（非同期HTTP通信、JSONフォーマット）
* サーバーサイド（想定仕様）
  * APIサーバー: Node.js, Python (FastAPI/Flask), Go等のWebフレームワーク
  * データベース: RDBMS（PostgreSQL, MySQL等）またはNoSQL/インメモリDB（Redis等）

## 3. 画面設計・UI/UX要件

### 3.1 ビューポートおよびレスポンシブ対応
* アスペクト比の固定: ゲームのプレイ領域（ステージ）は、スマートフォン縦画面を基準とした 9:16（基準解像度: 1080×1920）に固定する。
* 画面配置（センタリング）: PCやタブレットなどの横長ディスプレイで表示する際は、ゲームステージを画面の水平・垂直中央（Center-Center）に配置する。左右および上下の余白は、黒背景または背景グラフィックでマスク（カバー）する。
* スケーリング制御: CSSの object-fit: contain や、ゲームエンジンのScale Manager機能を活用し、デバイスの解像度に関わらず物理演算の座標系（ピクセル等価性）に矛盾が生じないよう、論理座標からデバイス座標への変換を正確に行う。

### 3.2 画面遷移とUIステート管理
本システムは単一のHTMLページ（SPA構成）内で、以下の4つのステート（状態）を管理・切り替えて表示する。

[1. タイトル画面]
- 主要UI要素: タイトルロゴ、ユーザー識別子（ユーザー名）入力フォーム、「ゲーム開始」ボタン、トップ10ランキングボード
- 説明: 起動時にサーバーからランキングを取得して表示。ユーザー名が未入力の場合はゲーム開始不可。

[2. ゲームプレイ画面]
- 主要UI要素: ゲームステージ（物理領域）、残り時間タイマー（バーまたは数値）、現在スコア、一時停止ボタン
- 説明: 物理演算がアクティブなメインゲーム画面。

[3. リザルト送信画面]
- 主要UI要素: 最終スコア表示、通信中インジケータ（スピンアニメーション）
- 説明: ゲーム終了後、スコアをAPIへ非同期送信し、レスポンスを待つ状態。

[4. ランキング画面]
- 主要UI要素: プレイヤーの最終順位、最新トップ10ランキングボード、「タイトルへ戻る」ボタン
- 説明: サーバーから返却された最新のランキングデータを反映して表示。

## 4. 機能要件

### 4.1 ゲームステージと物理挙動
* 物理ワールドの構築:
  - Matter.js等を用いて、ゲームステージの左右および下部に「不可視の壁（静的ボディ: isStatic: true）」を設定する。
  - 上部はオブジェクトの生成および初期落下スペースとするため、壁は配置しない。
* オブジェクトの定義:
  - 形状はすべて真円（Circle Body）とする。
  - デザイン（テクスチャ）は 5〜6種類 とし、ランダムに登場させる。
  - 物理パラメータ（質量、摩擦係数 friction、反発係数 restitution）を適切に調整し、オブジェクト同士の隙間に「滑り込む」「雪崩のように崩れる」リアルな挙動を実現する。
* 自動補充アルゴリズム:
  - ゲーム開始時、およびオブジェクト消去時、ステージ内の総オブジェクト数が規定数（例: 40〜50個）に達するまで、ステージ上方のランダムなX座標からオブジェクトを順次生成（Spawn）し、自由落下させる。

### 4.2 操作（イベントハンドリング）および消去ロジック
プレイヤーのドラッグ（スワイプ）操作による消去処理は、以下のイベントシーケンスに沿って完全な状態管理（ステートマシン）を行う。

1. 選択開始 (pointerdown / touchstart):
   - ポインタがオブジェクトの当たり判定（円の半径内）に入った場合、そのオブジェクトを「選択リスト（配列）」の1個目として登録する。
   - 選択されたオブジェクトにエフェクト（拡大や発光等）を付与する。
2. なぞり拡張 (pointermove / touchmove):
   - ポインタの移動中、以下の条件をすべて満たす隣接オブジェクトに触れた場合、選択リストに追加する。
     - 同種判定: 選択リストの1個目のオブジェクトと同一の種類（デザイン）であること。
     - 未選択判定: すでに選択リストに含まれていないこと。
     - 距離判定: 現在の選択リストの末尾（最後になぞった）オブジェクトとの中心間距離が、規定値（オブジェクトの直径の1.2倍以内など、物理的に隣接しているとみなせる距離）であること。
   - 取り消し（ロールバック）処理: ポインタが、選択リストの「末尾から2番目」のオブジェクトに戻った場合、現在の末尾オブジェクトを選択解除（リストから除外）し、なぞり直しの操作を許容する。
   - 選択中のオブジェクト間を結ぶ「ガイドライン（線）」をCanvas上に動的に描画する。
3. 選択確定・消去 (pointerup / touchend):
   - ポインタが画面から離れた際、選択リストの要素数が 3個以上 であれば消去処理へ移行する。2個以下の場合は、すべての選択を解除して通常状態に戻す。
   - 消去・スコア加算: 選択されたオブジェクトの物理ボディをワールドから削除（Composite.remove）する。
   - スコアは、消去数に応じたボーナス（例: 消去数 × 100 × コンボ倍率）を算出して加算する。
   - オブジェクトが消滅した空間に対し、上部にあったオブジェクトが物理演算に従って自然落下し、同時に上部から新たなオブジェクトが補充される。

### 4.3 ゲーム進行・ライフサイクル管理
* タイマー制御:
  - ゲーム開始と同時に、requestAnimationFrame を用いた高精度タイマー（デルタタイム考慮）により、60秒からのカウントダウンを行う。
* タイムアップ処理:
  - 残り時間が0になった瞬間、すべてのユーザー入力（イベントハンドリング）を即座に遮断する。
  - 物理演算のアップデートを一時停止（Freeze）させるか、またはステージ内の全オブジェクトが完全に静止するのを待ってからリザルト画面へ遷移する。

### 4.4 外部API・通信仕様
サーバーとの通信はすべて JSON 形式による非同期通信（HTTPS）で行う。

[① ランキング取得API]
- 用途: 起動時、またはゲーム終了時に全ユーザーのトップ10スコアを取得する。
- HTTPメソッド: GET
- エンドポイント: /api/ranking
- レンスポンス（JSON構造）:
  {
    "status": "success",
    "ranking": [
      { "rank": 1, "username": "PlayerA", "score": 12500, "achieved_at": "2026-06-20T12:00:00Z" },
      { "rank": 2, "username": "PlayerB", "score": 11000, "achieved_at": "2026-06-20T12:05:00Z" }
    ]
  }

[② スコア送信API]
- 用途: ゲーム終了時、獲得したスコアとユーザー識別子をサーバーに保存する。
- HTTPメソッド: POST
- エンドポイント: /api/score
- リクエスト（JSON構造）:
  {
    "username": "ユーザー入力文字列(最大15文字)",
    "score": 8500
  }
- レスポンス（JSON構造）:
  {
    "status": "success",
    "message": "Score saved successfully.",
    "your_rank": 5
  }

## 5. データモデル・データベース要件（参考）

### scores テーブル（リレーショナルDBを想定）
- id: BIGINT (PRIMARY KEY, AUTO_INCREMENT) -> レコードの一意識別子
- username: VARCHAR(15) (NOT NULL) -> プレイヤーのユーザー名
- score: INT (NOT NULL, DEFAULT 0) -> 獲得スコア
- created_at: TIMESTAMP (NOT NULL, DEFAULT CURRENT_TIMESTAMP) -> スコアが記録された日時

* インデックス設計:
  - ランキングを高速に取得するため、 score DESC, created_at ASC の複合インデックス（Composite Index）を推奨する。これにより、同点の場合は「先に高得点を達成したユーザー」を上位にするロジックを高速に処理できる。

## 6. 非機能要件

### 6.1 パフォーマンス要件
* フレームレート: モバイルおよびPCの主要ブラウザにおいて、オブジェクト最大表示時（50個程度）でも 安定して 60fps を維持すること。
* メモリ管理: オブジェクト消去時、Matter.jsのBodyインスタンスおよびPhaserのSpriteインスタンスを完全にメモリから解放（ガベージコレクションの対象に）すること。また、なぞり描画用のグラフィックオブジェクトは毎フレーム生成・破棄せず、プール（再利用）してメモリリークを防ぐ。

### 6.2 セキュリティ・信頼性
* 不正防止対策: フロントエンド側でスコアを確定させる仕様上、APIリクエストの偽装（不正スコアの直接POST）に対する考慮が必要である。将来的な拡張として、クライアント側から「消去ログのハッシュ値」をスコアと共に送信し、サーバー側で簡易的に整合性を検証するバリデーションの実装を推奨する。
* 通信エラーハンドリング: API通信がタイムアウトまたは500系エラー等で失敗した場合、ゲームデータをロストさせずに「再送信（Retry）ボタン」をリザルト画面に提示すること。

### 6.3 互換性要件
* 対応ブラウザ: iOS Safari (最新版), Android Chrome (最新版), Google Chrome, Microsoft Edge, Mozilla Firefox
* 入力互換: マウスイベント（mousedown等）とタッチイベント（touchstart等）の双方に個別対応するか、あるいは pointer イベント（pointerdown, pointermove, pointerup）へ統合して実装すること。
