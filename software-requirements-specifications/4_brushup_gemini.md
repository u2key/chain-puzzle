# ソフトウェア要求仕様書 (Software Requirements Specification)
## 物理演算ベース・なぞり消しパズルゲームシステム 【確定版】

---

## 1. システム概要
本システムは、スマートフォンの縦画面サイズを基準とし、PCやタブレットなどのマルチデバイスでもブラウザ上でプレイ可能な、物理演算を用いた落ち物パズルゲームである。
画面上部から降り注ぐ円形のオブジェクトを、プレイヤーが一筆書きのようになぞって3つ以上繋げて消去（チェーン）し、制限時間60秒以内での高得点を目指す。ゲーム終了後は、獲得スコアとユーザー識別子を非同期通信でサーバーに送信・保存し、ユーザーごとの最高スコアに基づいた最新のトップ10ランキングを表示する競争型のWebアプリケーションである。

---

## 2. システム構成・技術スタック
開発効率およびクロスプラットフォーム対応の最大化を目的とし、以下のWeb標準技術およびオープンソースライブラリを採用する。

* **クライアントサイド**
  * 言語: HTML5, CSS3, JavaScript (ES6以降)
  * ゲーム/物理エンジン: Matter.js（2D物理演算）および Phaser 3（またはCanvas APIを用いたカスタム描画フレームワーク）
  * 通信方式: Fetch API（非同期HTTP通信、JSONフォーマット）
* **サーバーサイド（想定仕様）**
  * APIサーバー: Node.js, Python (FastAPI/Flask), Go等のWebフレームワーク
  * データベース: RDBMS（PostgreSQL, MySQL等）またはNoSQL/インメモリDB（Redis等）

---

## 3. 画面設計・UI/UX要件

### 3.1 ビューポートおよびレスポンシブ対応
* **アスペクト比の固定:** ゲームのプレイ領域（ステージ）は、スマートフォン縦画面を基準とした 9:16（基準解像度: 1080×1920）に固定する。
* **画面配置（センタリング）:** PCやタブレットなどの横長ディスプレイで表示する際は、ゲームステージを画面の水平・垂直中央（Center-Center）に配置する。左右および上下の余白は、黒背景または背景グラフィックでマスク（カバー）する。
* **スケーリング制御:** CSSの `object-fit: contain` や、ゲームエンジンのScale Manager機能を活用し、デバイスの解像度に関わらず物理演算の座標系（ピクセル等価性）に矛盾が生じないよう、論理座標からデバイス座標への変換を正確に行う。

### 3.2 画面遷移とUIステート管理
本システムは単一のHTMLページ（SPA構成）内で、以下の4つのステート（状態）を管理・切り替えて表示する。

1. **タイトル画面:** ロゴ、ユーザー識別子入力フォーム、「ゲーム開始」ボタン、トップ10ランキングボードを表示。起動時にサーバーからランキングを取得して表示する。ユーザー名未入力時は開始不可。
2. **ゲームプレイ画面:** ゲームステージ（物理領域）、残り時間タイマー、現在スコア、一時停止ボタンを表示。物理演算がアクティブなメイン画面。
3. **リザルト送信画面:** 最終スコア表示、通信中インジケータ（スピンアニメーション）を表示。スコアをAPIへ非同期送信し、レスポンスを待つ。
4. **ランキング画面:** プレイヤーの最終順位、最新トップ10ランキングボード、「タイトルへ戻る」ボタンを表示。

## 4. 機能要件

### 4.1 ゲームステージと物理挙動

#### 4.1.1 物理ワールドの構築
Matter.js等を用いて、ゲームステージの左右および下部に「不可視の壁（静的ボディ: `isStatic: true`）」を設定する。上部はオブジェクトの生成および初期落下スペースとするため、壁は配置しない。

#### 4.1.2 オブジェクトの定義と物理パラメータ
ステージ内に登場するオブジェクトは、以下の基準に従って完全な同一規格の「真円（Circle Body）」として生成する。

| 項目 | 仕様定義値 |
| :--- | :--- |
| **オブジェクトの種類数** | 5種類固定（アセット名: `gem_1` ～ `gem_5`） |
| **出現確率** | 各種類一律均等（各20%の確率でランダム生成） |
| **サイズ（論理座標）** | 半経: 50 px （直径 100 px） ※基準解像度 1080×1920 における値 |
| **最大存在個数** | ステージ内 **最大50個固定** |
| **密度 (density)** | `0.001` （デフォルト値。質量は体積に比例して自動計算） |
| **摩擦係数 (friction)** | `0.05` （オブジェクト同士が隙間に滑り込みやすくするため低めに設定） |
| **空気抵抗 (frictionAir)** | `0.01` |
| **反発係数 (restitution)** | `0.2` （適度な弾みを持たせ、不自然に跳ねすぎない値に調整） |

#### 4.1.3 初期配置および自動補充アルゴリズム
* **初期配置時の制御（自動消滅の防止）:** ゲーム開始時、ステージ内にオブジェクトが50個に達するまで上方のランダムなX座標からオブジェクトを順次生成し、自由落下させる。この初期配置のタイミングに限り、同種オブジェクトが最初から3つ以上隣接して配置されても消去処理は実行しない。プレイヤーが最初のなぞり操作（`pointerdown`）を開始した瞬間の配置を初期状態としてゲームを開始する。
* **補充のタイミングとルール:** プレイヤーがポインタを離し（`pointerup`）、オブジェクトの消去およびそれに伴う消去エフェクト（アニメーション）が完全に完了した瞬間に、現在のステージ内の総オブジェクト数をカウントする。不足している数量（50個 － 残数）を計算し、その瞬間にまとめて上部のランダムなX座標から一斉に生成（Spawn）し、自由落下させる。なぞり操作の途中や、オブジェクトが消滅している最中のリアルタイム補充は行わない。

#### 4.1.4 アンチ・スタック（詰まり防止）システム
物理エンジンの噛み合わせや摩擦により、オブジェクトが空中で引っかかり、下部にスペースがあるにもかかわらず落下してこない現象（スタック）を防ぐため、以下のロジックを常時バックグラウンドで実行する。
* **スタック判定:** 生成後3秒以上経過したオブジェクトのうち、速度（`velocity`）が `0.1` 未満であり、かつ画面の底面や他の完全に静止したオブジェクトの山に到達していない（Y座標が不自然に高い位置で静止している）個体を検出する。
* **救済措置:** 該当するオブジェクトに対し、Matter.jsの `Body.applyForce` を用いて、ごく微小な横方向のランダムな力（フォース）を瞬間的に加え、噛み合わせを外して自然な落下を促す。

### 4.2 操作（イベントハンドリング）および消去ロジック
プレイヤーのドラッグ（スワイプ）操作による消去処理は、以下のイベントシーケンスに沿って完全な状態管理（ステートマシン）を行う。

1. **選択開始 (pointerdown / touchstart):** ポインタがオブジェクトの当たり判定（円の半径内）に入った場合、そのオブジェクトを「選択リスト（配列）」の1個目として登録する。選択されたオブジェクトにはエフェクト（拡大や発光等）を付与する。
2. **なぞり拡張 (pointermove / touchmove):** ポインタの移動中、以下の条件をすべて満たす隣接オブジェクトに触れた場合、選択リストに追加する。
   * **同種判定:** 選択リストの1個目のオブジェクトと同一の種類（デザイン）であること。
   * **未選択判定:** すでに選択リストに含まれていないこと。
   * **距離判定:** 現在の選択リストの末尾（最後になぞった）オブジェクトとの中心間距離が、オブジェクトの直径の1.2倍以内（120px以内）であること。
   * **取り消し（ロールバック）処理:** ポインタが、選択リストの「末尾から2番目」のオブジェクトに戻った場合、現在の末尾オブジェクトを選択解除（リストから除外）し、なぞり直しの操作を許容する。
   * 選択中のオブジェクト間を結ぶ「ガイドライン（線）」をCanvas上に動的に描画する。
3. **選択確定・消去 (pointerup / touchend):** ポインタが画面から離れた際、選択リストの要素数が3個以上（チェーン数 $N \ge 3$）であれば消去処理へ移行する。2個以下の場合は、すべての選択を解除して通常状態に戻す。

#### 4.2.3 スコア加算ロジック
1回のなぞり消し（チェーン）によって獲得できるスコアは、以下の計算式によって一意に算出する。算出に際し、小数点以下の値が発生した場合は、**一律切り捨て（Floor）**とする。なお、仕様書内において「コンボ」という概念・用語は一切排除し、「チェーン」に統一する。

$$Score = \lfloor N \times 100 \times (1 + (N - 3) \times 0.1) \rfloor$$

※ $N$ は1回のなぞり操作で消去されたオブジェクトの総数（チェーン数、$N \ge 3$）

* **3個消し（最小チェーン）:** $3 \times 100 \times (1 + 0) = \mathbf{300 \text{ 点}}$
* **4個消し:** $4 \times 100 \times (1 + 0.1) = \mathbf{440 \text{ 点}}$
* **5個消し:** $5 \times 100 \times (1 + 0.2) = \mathbf{600 \text{ 点}}$
* **10個消し（ロングチェーン）:** $10 \times 100 \times (1 + 0.7) = \mathbf{1,700 \text{ 点}}$

### 4.3 ゲーム進行・ライフサイクル管理
* **タイマー制御:** ゲーム開始と同時に、`requestAnimationFrame` を用いた高精度タイマーにより、60秒からのカウントダウンを行う。
* **タイムアップ処理:** 残り時間が0になった瞬間、すべてのユーザー入力（イベントハンドリング）を即座に遮断する。ステージ内の全オブジェクトが完全に静止するのを待ってからリザルト画面へ遷移する。

#### 4.3.4 一時停止（ポーズ）時の挙動と画面マスク
プレイヤーが「一時停止ボタン」をタップした際、ゲームステートを「ポーズ状態」に移行し、以下の制御を厳密に実行する。
* **タイマーの一時停止:** 60秒のカウントダウンタイマーをそのフレームで完全に停止する。
* **物理演算の完全停止:** Matter.jsのアップデートループ（`Engine.update`）を一時停止し、オブジェクトの落下や動きをすべてその場でフリーズさせる。
* **操作の遮断とリセット:** プレイヤーによるなぞり操作（ポインタイベント）を一切受け付けないようにイベントリスナーを無効化する。すでにオブジェクトをなぞっている最中（ドラッグ中）だった場合は、そのなぞりリストを強制的に全解除（ロールバック）して通常状態に戻してからポーズする。
* **画面のマスク（カンニング防止）:** ポーズ中はステージ全体の描画を隠すため、不透明または半透明の「ポーズUIオーバーレイ」を最前面に表示する。これにより、一時停止中にじっくり配置を見て次のルートを熟考する「カンニング行為」を完全に防止する。オーバーレイ上には「ゲームに戻る（再開）」ボタンを配置し、タップされたら上記すべての制限を解除してゲームを再開する。

### 4.4 外部API・通信仕様
サーバーとの通信はすべて JSON 形式による非同期通信（HTTPS）で行う。

**① ランキング取得API**
* HTTPメソッド: GET
* エンドポイント: `/api/ranking`
* レスポンス（JSON構造）:
  ```json
  {
    "status": "success",
    "ranking": [
      { "rank": 1, "username": "PlayerA", "score": 12500, "achieved_at": "2026-06-20T12:00:00Z" },
      { "rank": 2, "username": "PlayerB", "score": 11000, "achieved_at": "2026-06-20T12:05:00Z" }
    ]
  }
  ```

**② スコア送信API**
* HTTPメソッド: POST
* エンドポイント: `/api/score`
* リクエスト（JSON構造）:
  ```json
  {
    "username": "ユーザー名(1〜15文字)",
    "score": 8500
  }
  ```
* レスポンス（JSON構造）:
  ```json
  {
    "status": "success",
    "message": "Score saved successfully.",
    "your_rank": 5
  }
  ```

#### 4.4.3 APIバリデーションおよびエラー応答仕様
* **ユーザー名（username）のバリデーション規則:** 文字数は **1文字以上、最大15文字まで** とする（空文字やスペースのみの構成は不可）。半角英数字、ひらがな、カタカナ、漢字、主要記号を許可する。ただし、XSSやSQLインジェクション対策として、HTMLタグ（`< >`）や制御文字の入力はフロント・バックエンド双方でバリデーションを行い、弾くものとする。
* **通信エラー・タイムアウト処理:** API通信のタイムアウト時間は **5000ms（5秒）** とする。タイムアウト、またはサーバーエラー（HTTPステータス 400, 500系等）が発生した場合、ゲームデータをロストさせずに「送信に失敗しました」の文言とともに**「再送信（Retry）ボタン」**をリザルト画面に提示し、ユーザーが手動で再試行できるようにする。
* **エラーレスポンス（JSON構造例 / 400 Bad Request等）:**
  ```json
  {
    "status": "error",
    "message": "Invalid username. Spaces only or HTML tags are not allowed."
  }
  ```

#### 4.4.4 ランキング順位算出規則（重複排除）
データベースからトップ10を抽出・表示する際は、以下のルールを厳密に適用する。
* **同一ユーザーの複数記録の扱い（重複排除）:** 同一ユーザー（同一の `username`）が複数回スコアを送信した場合、**ランキング（トップ10）にはそのユーザーの「最高スコア（自己ベスト）」の1件のみを抽出**して表示する。1人のプレイヤーがランキング内に複数ランクインすることは一律不可とする。
* **同点時の順位決定（タイブレーク）:** 異なるユーザー間でスコアが完全に同点だった場合は、**「先にそのスコアを達成した（`created_at` が古い）ユーザー」を上位**とする。

## 5. データモデル・データベース要件

### scores テーブル（リレーショナルDBを想定）
* `id`: BIGINT (PRIMARY KEY, AUTO_INCREMENT) -> レコードの一意識別子
* `username`: VARCHAR(15) (NOT NULL) -> プレイヤーのユーザー名
* `score`: INT (NOT NULL, DEFAULT 0) -> 獲得スコア
* `created_at`: TIMESTAMP (NOT NULL, DEFAULT CURRENT_TIMESTAMP) -> スコアが記録された日時

#### データ保持方針とランキング抽出クエリ
ユーザーの過去のプレイ履歴（スコアの変遷）を保持するため、スコア送信時はレコードを上書きせず、毎回新しいレコードとしてインサートする。ランキング表示のトップ10を抽出する際は、各ユーザーの最高スコアのみをグループ化して抽出し、同点先着順を適用する以下のSQLを使用する。

```sql
SELECT username, MAX(score) AS max_score, MIN(created_at) AS achieved_at
FROM scores
GROUP BY username
ORDER BY max_score DESC, achieved_at ASC
LIMIT 10;
```

#### インデックス設計
上記の `GROUP BY` および `ORDER BY` クエリを高速に処理するため、**`username`, `score`, `created_at` による複合インデックス（Composite Index）**を構築し、データベースの走査パフォーマンスを最適化する。

## 6. 非機能要件

### 6.1 パフォーマンス要件
* **フレームレート:** モバイルおよびPCの主要ブラウザにおいて、オブジェクト最大表示時（50個固定）でも安定して **60fps** を維持すること。
* **メモリ管理:** オブジェクト消去時、Matter.jsのBodyインスタンスおよびPhaserのSpriteインスタンスを完全にメモリから解放（ガベージコレクションの対象に）すること。また、なぞり描画用のグラフィックオブジェクトは毎フレーム生成・破棄せず、プール（再利用）してメモリリークを防ぐ。

### 6.2 セキュリティ・信頼性
* **不正防止対策:** クライアント側から「消去ログのハッシュ値」をスコアと共に送信し、サーバー側で簡易的に整合性を検証するバリデーションの実装を推奨する。
* **通信信頼性:** 4.4.3節に規定した通り、5秒のタイムアウトおよび再送信（Retry）機能を具備すること。

### 6.3 互換性要件
* **対応ブラウザ:** iOS Safari (最新版), Android Chrome (最新版), Google Chrome, Microsoft Edge, Mozilla Firefox
* **入力互換:** マウスイベントとタッチイベントの双方に個別対応するか、あるいは pointer イベント（`pointerdown`, `pointermove`, `pointerup`）へ完全に統合して実装すること。
