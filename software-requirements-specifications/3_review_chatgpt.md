# SRSレビュー報告書
## 対象
物理演算ベース・なぞり消しパズルゲームシステム 要求仕様書

## 総合評価
**評価: 7.5 / 10**

本仕様書はゲーム概要、UI、基本ロジック、API、DB設計まで一通り記載されており、プロトタイプ開発は可能です。

ただし、「ゲーム性を左右するルール」「実装者ごとに解釈が分かれる部分」「サーバーとの契約仕様」が複数箇所で暗黙仕様のまま残っています。

現状は「設計仕様書寄り」であり、「誰が実装しても同じゲームになるレベルのプログラミング可能なSRS」としては不足があります。

---

# 良い点

- システム概要が明確
- 画面遷移が定義されている
- 操作フローが時系列で整理されている
- APIのJSON例がある
- DBモデルのたたき台がある
- パフォーマンス要件がある
- 不正対策や通信エラーへの考慮がある

---

# 重大な不足事項

## 1. オブジェクト仕様が曖昧

現状:

- 5〜6種類
- ランダム出現

のみ記載。

不足:

- 正確な種類数
- 出現確率
- 半径
- スプライトサイズ
- 物理パラメータ

例:

```text
ObjectType = 6種類固定

半径: 48px

出現率:
A=20%
B=20%
C=20%
D=15%
E=15%
F=10%
```

実装者ごとにゲームバランスが変わるため必須です。

---

## 2. スコア計算式が未定義

現状:

```text
消去数 × 100 × コンボ倍率
```

のみ。

不足:

- コンボ倍率計算式
- コンボ継続条件
- コンボリセット条件
- 小数切捨て方法

例:

```text
score =
deleteCount × 100 × comboMultiplier

comboMultiplier =
1.0 + (comboCount × 0.2)

最大倍率 5.0
```

---

## 3. コンボの存在が暗黙仕様

本文中で突然登場しています。

しかし

- コンボ定義
- コンボ発生条件
- 表示方法

が存在しません。

実装不能です。

---

## 4. ゲームオーバー条件が曖昧

仕様では

```text
60秒で終了
```

のみ。

未定義:

- オブジェクトが上端を超えた場合
- スポーン不能になった場合
- 物理エンジン暴走時

時間切れ以外では終了しないのか明記が必要です。

---

## 5. 補充アルゴリズムが不完全

未定義:

- 一度に何個補充するか
- 補充周期
- スポーン衝突時の処理
- 初期配置方法

例:

```text
目標数50個

消去後:
不足数を即時補充
```

# API仕様の不足

## 1. エラー応答がない

必要:

```json
{
  "status":"error",
  "message":"Invalid username"
}
```

### HTTPステータス

- 200
- 400
- 404
- 429
- 500

を定義するべきです。

---

## 2. バリデーション不足

ユーザー名について

```text
最大15文字
```

のみ。

必要:

- 最小文字数
- 空白許可可否
- 禁止文字
- Unicode許可範囲

例:

```text
1～15文字
英数字・日本語のみ
HTMLタグ禁止
```

---

## 3. ランキング仕様不足

未定義:

- 同点順位処理
- 同一ユーザー複数登録可否
- 最新記録優先か最高記録優先か

必須です。

# 結論

本仕様書はプロトタイプ開発には十分な品質です。

しかし「実装者が変わっても同一ゲームが完成するレベル」のSRSとしては、ゲームルール・スコア計算・物理パラメータ・ランキング仕様の定義が不足しています。

優先順位としては以下を推奨します。

1. スコア／コンボ仕様の完全定義
2. オブジェクト仕様の数値化
3. ランキング仕様の厳密化
4. APIエラー仕様の追加
5. 一時停止とゲーム終了条件の明文化

これらを追加すれば、実装可能性と保守性が大幅に向上します。
